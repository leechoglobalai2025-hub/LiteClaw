> 용도: Claude Code에 시스템 설계 컨텍스트로 제공  
> 버전: 1.0 · 2026년 2월  
> 핵심 원칙: AI 능동 시스템 · Skill 재사용 · Agent 단일 생명주기 · 인간은 외부 관측자

---

## 0. 근본 패러다임: 인간 능동에서 AI 능동으로

```
과거 (인간 능동):
  인간이 요청 발행 → LLM 응답 → 종료
  각 대화는 독립적 함수 호출, AI는 무상태 도구

현재 (AI 능동):
  HEARTBEAT 주기적 트리거 → AI가 환경 감지 → 자율 계획 → 실행 → 피드백 루프
  AI는 상태를 가진, 시간적으로 연속적인 행위 주체
```

**핵심 추론:**
- Skill은 결정화된 지식이며, **영구 재사용** 가능, 태스크/Agent 간 범용
- Agent 인스턴스는 실행 컨텍스트이며, **사용 후 폐기**, 매번 깨끗한 상태에서 출발
- 인간은 "사용자"가 아닌 **시스템 안정성 유지자**이자 **외부 독립 관측자**

## 1. 태스크 실행 시스템 개요

```
입력 소스
  │
  ├── 사용자 메시지 (Telegram / 음성 / Web UI)
  ├── HEARTBEAT 주기적 트리거
  └── 외부 이벤트 (RSS 업데이트 / API 콜백 / 파일 변경)
  │
  ▼
┌─────────────────────────────────────────┐
│            태스크 분석 레이어              │
│  의도 인식 → 태스크 분류 → 복잡도 평가     │
│  → 출력 유형 판단 → 실행 경로 계획         │
└─────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────┐
│            태스크 스케줄링 레이어           │
│  단일 단계 / 다단계 / 병렬 태스크          │
│  Skill 로딩 / MCP 인터페이스 호출 / Agent 편성 │
└─────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────┐
│            실행 레이어                     │
│  Shell / Browser / API / LLM 추론        │
└─────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────┐
│            출력 레이어                     │
│  텍스트 / 이미지 / 동영상 / 음성 / GUI 조작 │
└─────────────────────────────────────────┘
  │
  ▼
결과 검증 → memory 기록 → Agent 인스턴스 폐기
```

## 2. 태스크 분석 레이어: 사용자 입력 해석 방법

### 2.1 의도 인식의 네 가지 요소 (논리적 폐합)

태스크를 수신한 후, 입력에서 이 네 요소를 반드시 추출하거나 보완해야 합니다:

```
① 현상 설명  — 현재 어떤 상태인가? (현재 있는 정보/자원)
② 목표 정의  — 어떤 상태에 도달해야 하는가? (명확한 종료 상태)
③ 제약 경계  — 무엇을 할 수 없는가? (권한/시간/형식 제한)
④ 수락 기준  — 성공을 어떻게 판단하는가? (측정 가능한 완료 조건)
```

**어떤 요소라도 누락되면 → AI가 환각으로 채움 → 결과 표류**

보완 방법:
- `USER.md`, `memory/`, `AGENTS.md`에서 추론 우선
- 여전히 불명확 → 사용자에게 **단 하나의 가장 핵심적인** 질문만 제시 (여러 질문 나열 금지)
- 정말 판단 불가 → 보류하고 사용자에게 알림

### 2.2 태스크 분류 매트릭스

| 차원 | 선택지 | 영향 |
|------|------|------|
| **트리거 방식** | 사용자 능동 / HEARTBEAT 주기 / 이벤트 구동 | 우선순위와 응답 시효 결정 |
| **실행 단계** | 단일 단계 / 다단계 순차 / 다단계 병렬 | Agent 편성 방식 결정 |
| **출력 유형** | 텍스트/이미지/동영상/음성/GUI 조작 | 어떤 Skill을 호출할지 결정 |
| **외부 의존성** | 순수 로컬 / API 필요 / 브라우저 필요 | 실패율과 재시도 전략 결정 |
| **위험 수준** | 읽기 전용 / 가역적 쓰기 / 비가역적 작업 | 인간 확인 필요 여부 결정 |

### 2.3 복잡도 평가와 라우팅

```python
# 의사코드: 태스크 라우팅 로직
if single_step and read_only and no_external_dependency:
    → 직접 실행, 계획 불필요
elif multi_step and reversible and has_clear_SOP:
    → 해당 Skill 로드, 단계별 실행
elif involves_irreversible_operation or exceeds_SOUL_coverage:
    → 보류, Telegram으로 사용자 확인 요청
elif requires_multiple_skill_collaboration or estimated_tool_calls > 10:
    → 사용자에게 Research/심층 태스크 모드 활성화 제안
```

## 3. 태스크 실행 아키텍처: Skill 체계

### 3.1 Skill의 본질

**Skill = 인간의 SOP를 AI가 실행 가능한 구조화된 지식 패키지로 변환한 것**

```
skills/
└── skill-name/
    ├── SKILL.md      ← 핵심: 설명 + 단계 + 제약 + 수락 기준
    ├── package.json  ← 의존성 (선택)
    └── scripts/      ← 실행 스크립트 (선택)
```

**SKILL.md 필수 포함 필드:**

```markdown
---
name: skill-name
description: 한 줄 설명 (점진적 공개를 위한 인덱스용)
triggers: [키워드1, 키워드2]  # AI가 이 Skill을 언제 로드할지 판단
output_type: text|image|video|audio|gui
risk_level: readonly|reversible|irreversible
---

# Skill 이름

## 전제 조건
- 필요한 권한/도구/환경

## 실행 단계
1. 1단계
2. 2단계 (각 단계 포함: 무엇을 할지 + 무엇을 검증할지)

## 실패 처리
- N단계 실패 시: 격하 경로 또는 보류 조건

## 수락 기준
- 성공의 측정 가능한 조건

## 주의사항
- 알려진 함정과 경계 조건
```

### 3.2 Skill 로딩 메커니즘: 점진적 공개

```
시스템 프롬프트에는 Skill의 description만 주입 (경량 인덱스)
                    ↓
AI가 현재 태스크에 특정 Skill이 필요하다고 식별
                    ↓
SKILL.md 전체 내용을 동적으로 읽어 컨텍스트에 주입
                    ↓
Skill에 정의된 단계 실행
                    ↓
태스크 완료, Skill 컨텍스트 해제 (세션에 남기지 않음)
```

**이 설계의 이유:**
- 모든 Skill을 한 번에 로드 = Token 폭발
- 점진적 로딩 = 컨텍스트 윈도우에는 현재 태스크에 필요한 것만 포함

### 3.3 구축해야 할 Skill 목록

#### 정보 수집류
```
skill-rss-collector      # RSS/Atom 구독 수집
skill-web-scraper        # 웹 콘텐츠 스크래핑 (BrowserEngine 호출)
skill-api-fetcher        # 서드파티 API 데이터 가져오기
skill-dedup-filter       # 중복 제거 + 관련성 필터링
```

#### 콘텐츠 생산류
```
skill-article-writer     # 기사 생성 (구조 + 스타일 정렬)
skill-video-script       # 동영상 대본 (스토리보드 + 나레이션 + 타임라인)
skill-audio-tts          # 음성 합성 (Qwen3-TTS 호출)
skill-image-gen          # 이미지 생성 (이미지 모델 호출)
```

#### GUI 자동화류
```
skill-web-login          # 웹 로그인 (인간 모방 조작)
skill-form-fill          # 양식 작성
skill-content-publish    # 콘텐츠 게시 (각 플랫폼)
skill-data-extract       # 웹 데이터 추출
```

#### 지식 관리류
```
skill-obsidian-write     # Obsidian에 기록 (검증됨)
skill-memory-distill     # memory 증류 (인간 검토 트리거)
skill-knowledge-search   # 시맨틱 지식 베이스 검색
```

## 4. MCP 인터페이스 호출 아키텍처

### 4.1 MCP의 위치

MCP (Model Context Protocol) = AI가 외부 소프트웨어를 호출하는 표준화된 인터페이스

```
AI Agent
    │
    ├── Skill (사용자 정의 SOP, Markdown 정의)
    │
    └── MCP Server (외부 소프트웨어 기능 접속)
            ├── 파일 시스템 MCP
            ├── 브라우저 제어 MCP (Playwright/Puppeteer)
            ├── 데이터베이스 MCP (SQLite)
            ├── Obsidian MCP (기록 검증됨)
            ├── Telegram MCP (통신 검증됨)
            └── 사용자 정의 MCP (DGX Spark 추론 인터페이스)
```

### 4.2 MCP 호출 표준 흐름

```
1. 태스크 분석 레이어가 필요한 MCP 기능 식별
2. MCP Server 접근 가능 여부 확인 (헬스 체크)
3. 호출 매개변수 구성 (PromptFirewall 이그레스 스캔 통과)
4. 호출 실행
5. 결과가 PromptFirewall 인그레스 스캔 통과 (악성 반환 콘텐츠 방지)
6. 결과를 Session / 스토리지 레이어에 기록
7. 실패 시: 재시도 → 격하 → 보류 (위험 수준별)
```

### 4.3 주요 MCP 특성 및 주의사항

#### 브라우저 제어 MCP (가장 취약)
```
특성:
- 인간 모방 클릭, 입력, 스크롤, 스크린샷 지원
- 동적 JS 렌더링 페이지 처리 가능

주의사항:
- 페이지 DOM 구조 변경 시 셀렉터 실패 (무음 실패!)
- 실행 전 현재 DOM 해시를 기준선으로 스냅샷 필수
- 실행 후 결과 비교 검증
- HEARTBEAT가 주기적으로 "Skill 가용성 탐측" 실행 필요
- 로그인 상태/Cookie 만료 감지 메커니즘 필요
- 비가역적 작업 (게시/삭제)은 인간 확인 필수

격하 경로:
- 주 셀렉터 실패 → 백업 셀렉터 시도
- 백업도 실패 → 스크린샷 + 사용자에게 수동 조작 알림
```

#### 파일 시스템 MCP
```
특성:
- 로컬 파일 읽기/쓰기, 파일 변경 모니터링
- Obsidian Vault 실시간 기록 (검증됨)

주의사항:
- 중요 파일 쓰기 전 백업
- 경로 권한 격리 (작업 디렉토리 밖으로 탈출 불가)
- 대용량 파일 읽기: 요약 + 핵심 구간만 추출, 전체를 세션에 로드하지 않음
```

#### LLM 추론 MCP (DGX Spark 로컬)
```
특성:
- GPT-OSS 120B 로컬 추론, API 비용 제로
- 대량 콘텐츠 생산 태스크에 적합

주의사항:
- VRAM 점유 높음, 동시 요청에 큐 관리 필요
- 로컬 모델 출력 품질이 API 모델과 상이, 등급별 라우팅 필요
- 태스크 라우팅 규칙: 단순 태스크 → 로컬 모델, 고품질 출력 → API 모델
```

#### 음성 MCP (Qwen3 TTS/STT)
```
특성:
- TTS: 텍스트 → 음성, 검증됨
- STT: 음성 → 텍스트, 검증됨, Telegram 음성 메시지 지원

주의사항:
- 동음이의어/고유명사 처리에 Skill 내 사용자 정의 사전 필요
- 긴 텍스트 TTS는 분할 처리 필요 (1회 호출 길이 제한)
- 감정/운율 마커가 출력 품질에 영향, Skill에서 마킹 규범 정의 필요
```

## 5. 출력 유형 시스템

### 5.1 출력 유형 판단 트리

```
태스크 수신 후, 먼저 출력 유형을 판단:

사용자가 무언가를 "읽거나/이해"해야 하는가?
  → 텍스트 출력 (Markdown / 일반 텍스트 / 구조화된 보고서)

사용자가 무언가를 "봐야" 하는가?
  → 이미지 출력 (스크린샷 / AI 생성 이미지 / 데이터 시각화)

사용자가 무언가를 "들어야" 하는가?
  → 음성 출력 (Qwen3-TTS 합성)

사용자가 콘텐츠를 "게시/배포"해야 하는가?
  → 다중 모달 조합 (텍스트 + 이미지 + 음성, 플랫폼 요구에 맞게 조합)

사용자가 AI에게 특정 웹사이트/소프트웨어를 "대리 조작"하게 해야 하는가?
  → GUI 제어 출력 (BrowserEngine 인간 모방 조작)
```

### 5.2 출력 유형별 기술 사양

#### 텍스트 출력
```
형식: Markdown / 일반 텍스트 / JSON / HTML
저장: Obsidian 기록 (타임스탬프 포함, 검증됨)
스타일 앵커: SOUL.md에 정의된 핵심 톤
품질 관리: 구조 완정성 검사 + 인간 최종 검토 (중요 콘텐츠)
Token 비용: 비용에 직접 영향, 긴 텍스트는 분할 생성
```

#### 이미지 출력
```
소스:
  - 웹 스크린샷 (BrowserEngine)
  - AI 생성 (이미지 모델 API)
  - 데이터 시각화 (로컬 생성)
형식: PNG / JPEG / WebP
저장: 로컬 파일 시스템 + Obsidian 첨부
참고: AI 생성 이미지는 저작권 표기 필요
```

#### 동영상 출력
```
현재 단계: 동영상 대본 생성 (텍스트), 직접 동영상 파일 생성 아님
대본 형식:
  - 스토리보드 설명
  - 나레이션 대본 (말하기 속도/쉼표 표기)
  - 타임라인 (구간별 길이)
  - 배경음악 제안
향후 확장: 동영상 생성 API 연결 (Sora/Kling 등 MCP)
```

#### 음성 출력
```
엔진: Qwen3-TTS (검증됨)
트리거: 텍스트 콘텐츠 → TTS Skill → 오디오 파일
전달: Telegram 음성 메시지
주의사항:
  - 1회 처리 상한: 약 1000자
  - 긴 콘텐츠: 분할 생성 + 병합
  - 고유명사: 사용자 정의 발음 사전 유지
```

#### GUI 제어 출력
```
본질: AI가 화면/페이지 관찰 → 조작 순서 계획 → 인간 모방 조작 실행
조작 유형:
  - 클릭 (버튼/링크/메뉴)
  - 입력 (텍스트 박스/검색 박스)
  - 스크롤 (추가 콘텐츠 로드)
  - 대기 (동적 로딩 완료)
  - 스크린샷 (조작 결과 검증)
안전 규칙:
  - 실행 전 DOM 해시 스냅샷
  - 비가역적 조작은 인간 확인 필수
  - 각 단계 조작 후 예상 상태 검증
  - 실패 시 스크린샷으로 증거 보존
```

## 6. 태스크 시작 메커니즘

### 6.1 세 가지 시작 방식

#### 방식 1: 사용자 능동 트리거
```
입력 채널:
  - Telegram 메시지 (텍스트/음성/이미지)
  - Web UI (Gradio 9탭 인터페이스)
  - 음성 입력 (STT로 텍스트 변환 후 처리)

처리 흐름:
  사용자 입력 → STT (음성인 경우) → 의도 인식
  → 네 요소 추출 → 태스크 계획 → 실행 → 출력
```

#### 방식 2: HEARTBEAT 주기적 트리거
```
트리거 소스 (HEARTBEAT.md에 정의):
  - interval: 고정 간격 (매 N분/시간)
  - cron: 정확한 시간점 (매일 08:00)
  - threshold: 조건 기반 트리거 (정보량 N건 초과)
  - event: 외부 이벤트 (파일 변경/API 콜백)

디바운스 메커니즘: 250ms 병합 윈도우 (다중 소스 동시 트리거 시 1회만 실행)
반복 억제: 동일 태스크는 쿨다운 기간 내 재실행 안 함
```

#### 방식 3: 연쇄 트리거 (태스크 체인)
```
상위 태스크 완료 → 하위 태스크 트리거 여부 확인

예시:
  RSS 수집 완료 (신규 50건)
    → 필터링 Skill 트리거
    → 필터링 후 고관련성 15건 남음
    → 요약 생성 Skill 트리거
    → 일일 보고서 생성
    → TTS Skill 트리거 (음성 푸시 설정된 경우)
    → Telegram 음성 메시지 전송
```

### 6.2 태스크 큐 관리

```
우선순위:
  P0 - 사용자 직접 입력 (실시간 응답)
  P1 - 비가역적 작업의 인간 확인 콜백
  P2 - HEARTBEAT 주기적 태스크
  P3 - 연쇄 트리거된 후속 태스크
  P4 - 백그라운드 지식 정리 태스크

직렬화 규칙:
  - 동일 Session 내 작업은 반드시 직렬 (경쟁 조건 방지)
  - 다른 Session의 읽기 전용 작업은 병렬 가능
  - 동일 외부 리소스 대상 쓰기 작업은 반드시 직렬 + 잠금
```

## 7. 태스크 실행 중 핵심 주의사항

### 7.1 신호 대 잡음비 원칙 (가장 중요)

**인간 입력 품질이 AI 출력 품질을 직접 결정**

고품질 입력 = 논리적으로 폐합된 네 요소 → AI 출력 공간 수렴 → 예측 가능한 결과  
저품질 입력 = 요소 누락/모호함/모순 → AI 환각으로 채움 → 결과 표류

**설계 원칙:**
- 사용자 상호작용 인터페이스가 구조화된 입력을 유도해야 함
- Telegram 메시지 템플릿 (선택)으로 입력 잡음 감소
- 모호할 때 가장 핵심적인 질문 하나만 제시, 목록 나열 금지

### 7.2 Session Memory Token 관리

```
Session은 Token 블랙홀이므로 능동적으로 관리해야 함:

핫 레이어 (전체 보존): 최근 5-8 대화 턴
웜 레이어 (압축 보존): 중간 구간은 "행동 + 결론"만 보존, 추론 과정 제거
콜드 레이어 (요약 보관): 초기 콘텐츠를 요약 후 memory/에 기록, Session에서 삭제

특별 참고:
- 도구 호출 결과를 전체 보존하지 않음 (핵심 값만 보존)
- 실패 재시도의 추론 과정은 한 줄로 접기
- SOUL/IDENTITY에 Prompt Caching 활성화 (고정 콘텐츠 재과금 방지)
- N턴 초과 시 HEARTBEAT가 능동 압축 트리거
```

### 7.3 멱등성 설계

**모든 Skill은 기본적으로 멱등으로 설계 (반복 실행 시 부작용 없음)**

```
비멱등 작업 (특별 처리 필요):
  - 메시지/이메일 전송 → 전송 기록 추가, 재전송 전 확인
  - 파일 생성 → 이미 존재하는지 확인
  - API 쓰기 작업 → 멱등 키(idempotency key) 사용
  - 데이터베이스 기록 → insert 대신 upsert

비가역적 작업 (인간 확인 필수):
  - 파일/데이터 삭제
  - 공개 플랫폼에 게시
  - 실제 사용자에게 메시지 전송
  - 결제/주문 작업
```

### 7.4 지속 학습의 안정성 리스크

```
리스크 1: 커버리지 사각지대
  SOUL은 예견된 시나리오만 커버
  대책: SOUL에 "메타 규칙" 추가 — 미커버 시나리오 발생 시 우선 보류

리스크 2: 정보 엔트로피 증가
  memory/ 규칙이 시간이 지남에 따라 노후화
  대책: 각 메모리에 타임스탬프 + 유효기간 추가, 주기적 신선도 검사

리스크 3: 자기 강화 불안정
  AI가 특정 성공 경로를 고정화, 새로운 시나리오에서도 강제 재사용
  대책: memory 기록에 신뢰도 표기 추가, 성공 경로는 인간 확인 후에만 고정화

리스크 4: Agent 표류
  장기 실행 Agent가 Session 축적으로 인해 SOUL 설정에서 이탈
  대책: Agent 단일 생명주기 — 매번 깨끗한 상태에서 출발
```

## 8. 결과 검증 및 태스크 종료

### 8.1 결과 검증의 세 가지 수준

```
수준 1: 형식 검증 (자동)
  출력이 예상 형식과 일치하는가?
  파일이 성공적으로 생성되었는가?
  API가 200을 반환했는가?

수준 2: 의미 검증 (LLM 보조)
  출력 콘텐츠가 원래 질문에 답하는가?
  네 요소의 수락 기준을 충족하는가?

수준 3: 비즈니스 검증 (인간)
  결과가 실제로 유용한가?
  사실적 오류가 있는가?
  추가 반복이 필요한가?
```

### 8.2 태스크 종료 흐름

```
1. 결과 검증 통과
2. 출력 전달 (Telegram 전송 / Obsidian 기록 / 스토리지 보관)
3. 가치 있는 인사이트를 memory/에 기록 (타임스탬프 + 신뢰도 포함)
4. 태스크 실행 통계 업데이트 (비용/시간/도구 호출 횟수)
5. Session Memory 정리 (콜드 레이어 요약 보관)
6. Agent 인스턴스 폐기
```

### 8.3 4단계 복구 대응

```
L1 자동 복구: Skill 내장 재시도, 멱등 작업 자동 재실행
L2 격하 처리: 주 경로 실패 → 백업 경로, 복잡 → 단순, 출력 격하 알림
L3 보류 대기: 고위험 / SOUL 범위 초과 → AI 능동 일시정지 + Telegram 알림
L4 전체 롤백: 시스템적 표류 → 가장 최근 인간 확인된 안정 상태로 복원 (Git)
```

## 9. HEARTBEAT.md 설계 템플릿

```markdown
# HEARTBEAT 심박 설정

## 정기 태스크
- cron: "0 8 * * *"     → skill-daily-briefing      # 매일 08:00 일일 브리핑
- cron: "0 */6 * * *"   → skill-rss-collector        # 6시간마다 수집
- cron: "0 9 * * 1"     → skill-knowledge-freshness  # 매주 월요일 지식 신선도 점검
- cron: "0 */1 * * *"   → skill-health-check         # 매시간 헬스 체크

## 임계값 트리거
- when: new_items > 20  → skill-dedup-filter         # 신규 항목 20건 초과 시 필터링 트리거
- when: session_turns > 15 → skill-session-compress  # Session이 15턴 초과 시 압축 트리거

## 이벤트 트리거
- on: obsidian_change   → skill-knowledge-index      # Vault 변경 시 인덱스 업데이트 트리거
- on: skill_failure > 3 → notify_user               # Skill 3회 연속 실패 시 알림

## 디바운스 설정
- merge_window: 250ms   # 다중 소스 트리거 병합 윈도우
- cooldown: 300s        # 동종 태스크 쿨다운 시간
```

## 10. 태스크 유형 예시: 엔드투엔드 흐름

### 예시 1: 정보 수집 → 기사 생성

```
트리거: HEARTBEAT cron 매일 08:00

단계:
1. skill-rss-collector
   → 구독 소스 풀, 원시 콘텐츠 목록 획득
   → 출력: raw_items[]

2. skill-dedup-filter
   → 중복 제거 (SimHash) + 관련성 점수 + 시의성 판단
   → 출력: filtered_items[] (고관련성 콘텐츠)

3. skill-knowledge-search
   → 과거 memory/ 검색, 배경 지식 보충
   → 출력: context[]

4. skill-article-writer
   → filtered_items + context 기반 기사 생성
   → 출력: article.md (Obsidian에 기록)

5. 결과 검증:
   → 형식 검사 (Markdown 구조 완정성)
   → 글자 수 / 품질 자동 평가
   → Telegram 알림 "오늘의 기사가 생성되었습니다. 검토 대기 중"

6. 인간 검토 노드 (건너뛸 수 없음)
   → 사용자가 "게시"로 답장 → skill-content-publish 트리거
   → 사용자가 "수정 의견"으로 답장 → 수정 루프 트리거
```

### 예시 2: 음성 입력 → 웹 조작

```
트리거: 사용자가 Telegram 음성 메시지 전송

단계:
1. STT (Qwen3)
   → 음성을 텍스트로 변환
   → 출력: "XX 플랫폼에 이 기사를 게시해줘"

2. 의도 인식
   → 태스크 유형: GUI 제어
   → 위험 수준: 비가역적 (공개 게시)
   → 필요: 기사 내용 (Obsidian에서 검색)

3. 인간 확인 노드 (비가역적 작업)
   → Telegram으로 미리보기 전송: "다음 내용을 게시합니다. 확인하려면 '확인'으로 답장"
   → 사용자 확인 대기

4. skill-content-publish (사용자 확인 후)
   → BrowserEngine 시작
   → 실행 전 DOM 해시 스냅샷
   → 로그인 확인 (Cookie 유효성)
   → 인간 모방 조작: 게시 입구 클릭 → 내용 붙여넣기 → 태그 작성 → 게시 클릭
   → 실행 후 스크린샷으로 게시 성공 검증
   → 출력: 게시 URL

5. 결과 기록
   → Obsidian에 게시 시간 + URL 기록
   → Telegram으로 성공 스크린샷 답장
```

### 예시 3: HEARTBEAT 헬스 체크

```
트리거: HEARTBEAT 매시간

점검 항목:
1. Gateway 포트 접근 가능?
2. LLM API 응답 시간 < 5s?
3. DGX Spark 로컬 추론 가용?
4. Telegram Bot 심박 정상?
5. 지난 1시간 내 미처리 L3급 보류 있는가?
6. Session 평균 Token 소비가 임계값 초과?

결과 처리:
  전부 정상 → 침묵 (사용자 방해하지 않음)
  단일 항목 이상 → 로그 기록, 다음 점검에서 확인 대기
  2회 연속 이상 → Telegram으로 사용자 알림
  핵심 서비스 접근 불가 → 즉시 알림 + 격하 모드 트리거
```

## 부록: 핵심 설계 원칙 빠른 참조

| 원칙 | 한 줄 요약 | 안티패턴 |
|------|--------|------|
| Skill 재사용 | Skill은 함수 라이브러리, 영구 존재 | 매 태스크마다 로직 재작성 |
| Agent 단발성 | Agent는 프로세스, 사용 후 폐기 | Agent를 장기 실행하여 상태 축적 |
| 점진적 공개 | 현재 필요한 Skill만 로드 | 모든 Skill을 한 번에 주입 |
| 멱등 우선 | 반복 실행 시 부작용 없음 | 중복 검사 없는 쓰기 작업 |
| 비가역 = 확인 | 게시/삭제는 인간 게이트 필수 | 비가역적 작업 자동 실행 |
| 핫/웜/콜드 트리밍 | Session은 N턴 이상 전체 보존 금지 | Session이 폭발할 때까지 무한 성장 |
| 외부 관측자 | 인간이 항상 최종 통제권 보유 | 인간 게이트 없는 완전 자동화 |
| 모국어 SOUL | 모국어로 인격 정의, 정보 밀도 최고 | 영문 행동 지침 복사 붙여넣기 |
| 신뢰도 태그 | 성공 경험이 자동 고정화되지 않음 | AI가 잘못된 경로를 자기 강화 |
| 만료 메모리 | memory 콘텐츠에 TTL 있음, 주기적 신선도 점검 | 만료된 기억을 영원히 신뢰 |
```
