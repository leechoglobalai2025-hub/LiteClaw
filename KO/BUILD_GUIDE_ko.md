## 이 가이드가 존재하는 이유

### 두 번의 실제 실패 (2026년 3월 28일)

**실패 #1:** AI에게 문서 1을 먼저 제공했습니다. AI는 즉시 자체 해석을 기반으로 코딩을 시작했습니다. 나중에 문서 2를 제공하자 AI는 모든 것을 처음부터 다시 만들었고, 35개의 Python 파일이 생성되었지만 서로 참조하지 않았습니다. 61개의 테스트가 통과했으나 레이어를 연결하는 Agent Loop이 없었습니다. 죽은 코드였습니다.

**실패 #2:** AI에게 문서 1과 2를 동시에 제공하고 "완전한 LiteClaw 소프트웨어를 프로그래밍하라"고 지시했습니다. AI는 6개의 병렬 Agent를 실행하여 모든 레이어를 동시에 구축했습니다. 결과: 실제 기능과 분리된 템플릿 기반 UI, 두 개의 아키텍처 레이어 누락 (L1 Channel, L4 LLM), 레이어 간 통합 없음. 저자의 평가: "코드 쓰레기 산."

**두 실패의 근본 원인:** AI가 문서 2에 정의된 순차적 태스크 카드 순서를 따르는 대신 자체 실행 전략을 선택했습니다.

## 권장 환경

| 환경 | 권장도 | 이유 |
|:---|:---|:---|
| **IDE** (Google IDX, Cursor, Windsurf 등) | ✅ **권장** | 단일 Agent 실행, 시각적 파일 트리, 실시간 피드백, 자연스러운 순차 실행 |
| **CLI** (Claude Code, 터미널 기반 Agent) | ⚠️ **주의하여 사용** | 다중 Agent 병렬, 블랙박스 실행, 완료 후에야 결과 확인 가능 |

**최초의 무결 빌드는 Google IDE + Claude Opus 4.5 환경에서 달성되었습니다.** IDE 환경은 AI를 단일 스레드 순차 실행으로 자연스럽게 제약합니다 — 이 아키텍처가 요구하는 바로 그것입니다.

두 번의 CLI 실패는 CLI Agent가 병렬 하위 Agent를 자유롭게 생성할 수 있기 때문에 발생했습니다. AI는 "시스템 구축"을 "모든 것을 동시에 구축"으로 해석했고, 이는 레이어별 통합 원칙을 위반했습니다.

**CLI를 반드시 사용해야 한다면:** 아래의 시작 프롬프트에 병렬 방지 제약이 포함되어 있지만, IDE 수준의 강제 실행이 프롬프트 수준의 제약보다 본질적으로 더 안정적입니다.

## 올바른 시작 프로토콜

### 1단계: 두 문서를 모두 컨텍스트로 로드

문서 1과 문서 2를 **동시에** 같은 세션에서 AI에게 제공합니다. 아직 코딩을 시작하지 마세요.

### 2단계: 다음 시작 프롬프트를 정확히 사용

다음 프롬프트를 복사하여 붙여넣으세요:

```
LiteClaw라는 프로젝트의 두 개의 아키텍처 문서를 제공합니다.

문서 1 (아키텍처 블루프린트)은 무엇을 만들지 정의합니다.
문서 2 (태스크 카드 시스템)은 어떻게 만들지 정의합니다.

핵심 규칙 — 이 중 하나라도 위반하면 손상된 코드가 생성됩니다:

1. 순차 실행만 허용: 태스크 카드를 엄격한 순서로 실행하세요:
   TASK-00 → TASK-01 → TASK-02 → ... → TASK-28.
   절대 건너뛰지 마세요. 절대 병렬화하지 마세요. 절대 여러 태스크를 한꺼번에 처리하지 마세요.

2. 태스크당 하나의 세션: 각 태스크 카드 = 하나의 독립 코딩 세션.
   하나의 태스크를 완전히 완료한 후 멈추고, 무엇을 만들었는지 보고하세요.
   다음 태스크로 진행하기 전에 내 검증을 기다리세요.

3. 레이어 통합: 각 새 태스크는 이전에 완료된 태스크의 코드를 반드시 import하고 참조해야 합니다.
   고립된 모듈은 허용되지 않습니다. 모든 새 파일은 기존 코드베이스에 연결되어야 합니다.

4. 템플릿 투기 금지: 보일러플레이트나 플레이스홀더 코드를 생성하지 마세요.
   모든 코드 라인은 태스크 카드에 명시된 실제 기능을 구현해야 합니다.
   태스크 카드에 "최대 80줄"이라고 되어 있으면 그것을 준수하세요.

5. 진행 전 검증: 각 태스크 후 지정된 테스트를 실행하세요.
   보고: 생성된 파일, 코드 줄 수, 테스트 결과, 이전 레이어와의 통합 포인트.
   그런 다음 내 확인을 기다리세요.

6. 재구성 금지: 태스크 카드에 명시된 정확한 디렉토리 구조와 파일명을 따르세요.
   이름 변경, 재구조화, 조직 "개선"을 하지 마세요.

지금 TASK-00부터 시작하세요. 무엇을 만들었는지 보여주세요. TASK-00 완료 후 멈추고
내 검증을 기다리세요.
```

### 3단계: 각 태스크를 진행 전에 검증

각 태스크 후 확인:

```
□ 생성된 파일이 태스크 카드 사양과 일치하는가?
□ 코드 줄 수가 태스크 카드 제한 내인가?
□ 명시된 모든 테스트가 통과하는가?
□ 새 코드가 이전 레이어 코드를 import/참조하는가?
□ 플레이스홀더나 스텁 함수가 없는가?
```

모두 통과하면 AI에게: **"TASK-XX 검증 통과. TASK-XX+1로 진행하세요."**

하나라도 실패하면: **"TASK-XX 검증 실패. 새 세션을 열고 TASK-XX만 다시 하세요."**
같은 세션에서 수정하게 하지 마세요 — 엔트로피가 누적됩니다.

### 4단계: 페이즈 체크포인트

각 Phase 종료 시 전체 통합 테스트 수행:

```
Phase 0 (TASK-00):       pip install -e . 작동, 원본 nanobot 기능 사용 가능
Phase 1 (TASK-01~04):    보안 레이어가 Agent Loop에 통합, 키 절대 미노출
Phase 2 (TASK-05~09):    라우터가 태스크 레벨별 모델 선택, 장애 전환 작동
Phase 3 (TASK-10~13):    컨텍스트 레이어가 기존 context.py를 대체, Token 예산 적용
Phase 4 (TASK-14~16):    감사 엔진이 파괴적 작업 차단, Obsidian에 로그 기록
Phase 5 (TASK-17~20):    실행 엔진이 통합 ExecutionManager를 통해 연결
Phase 6 (TASK-21~23):    모든 데이터가 SQLite에 영속화, 비용 추적 정확
Phase 7 (TASK-24~28):    UI가 실제 컴포넌트에 연결, 전체 5개 탭 기능 정상
```

## 안티패턴 (하지 말아야 할 것들)

| ❌ 안티패턴 | 실패 이유 | ✅ 올바른 접근 |
|:---|:---|:---|
| "완전한 시스템을 구축하라" | AI가 모든 것을 병렬화, 연결되지 않는 모듈 생성 | "TASK-00만 시작하세요. 멈추고 기다리세요." |
| 문서 1을 먼저, 문서 2를 나중에 | AI가 시공 도면을 보기 전에 코딩 시작 | 두 문서를 함께 제공한 후 시작 프롬프트 사용 |
| AI가 아키텍처를 "개선"하게 허용 | AI가 파일 재구성, 클래스 이름 변경, 프레임워크 추가 | "정확한 사양을 따르세요. 수정 없이." |
| 같은 세션에서 실패한 태스크 수정 | 세션 컨텍스트가 잘못된 코드로 오염, 엔트로피 증가 | 새 세션에서 실패한 태스크를 처음부터 다시 수행 |
| 태스크 간 검증 건너뛰기 | N+1 레이어가 결함 있는 N 레이어 위에 구축, 오류 연쇄 | 모든 태스크를 검증. 예외 없음. |
| 여러 병렬 Agent 사용 | 각 Agent가 격리되어, 통합되지 않는 코드 생성 | 단일 순차 Agent. 한 번에 하나의 태스크. |

## 검증 결과 (2026년 3월 28일)

### 세 차례 테스트, 같은 날, 같은 CLI 환경, 같은 Opus 4.6:

| 라운드 | 시작 지시 | 실행 방식 | 결과 |
|:---|:---|:---|:---|
| **1라운드** | 문서 1을 먼저, 문서 2를 나중에 제공 | AI가 자체 순서 결정 | ❌ 61개 테스트 통과, 모듈 고립, Agent Loop 없음 |
| **2라운드** | "완전한 LiteClaw 소프트웨어를 프로그래밍하라" | 6개 병렬 Agent | ❌ 템플릿 UI, 레이어 누락, 코드 쓰레기 산 |
| **3라운드** | **"절대 다중 Agent 금지! 한 단계씩!"** + 순차 실행 강제 | **엄격한 TASK-00→28, 단일 Agent** | ✅ **227개 테스트 통과, 전체 8개 레이어 통합, 완성 제품** |

**같은 문서. 같은 모델. 같은 환경. 오직 시작 지시만 바뀌었습니다.**

### 3라운드 검증 지표:

- **29장 태스크 카드:** 전부 엄격한 순서로 완료
- **227개 테스트:** 전부 통과, 페이즈 간 제로 회귀
- **8개 아키텍처 레이어:** L0 Security → L7 Audit, Agent Loop을 통한 완전 통합
- **페이즈 체크포인트:** 다음 페이즈로 진행 전 모든 페이즈 검증
- **테스트 누적 궤적:** 6 → 34 → 80 → 91 → 118 → 167 → 196 → 227 (단조 증가)

### 역사적 검증:

- **최초 빌드 (2026년 2월):** Google IDE + Claude Opus 4.5 → 약 5시간, 거의 무결
- **CLI 빌드 (2026년 3월 28일):** Claude Code CLI + Claude Opus 4.6 → 전체 29개 태스크 완료, 227개 테스트 통과
