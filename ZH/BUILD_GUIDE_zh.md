## 为什么需要这份指南

### 两次真实失败（2026年3月28日）

**失败 #1：** 先给 AI 喂了文档1。AI 立即按照自己的理解开始编码。后来补充文档2时，AI 推翻重建——生成了35个 Python 文件，全部存在但互不引用。61个测试通过了，但系统没有连接各层的 Agent Loop。死代码。

**失败 #2：** 同时给 AI 提供文档1和文档2，指令是"编程出完整的 LiteClaw 软件"。AI 启动了6个并行 Agent 同时构建所有层。结果：模板化 UI 与实际功能完全脱节，缺失两个完整的架构层（L1 Channel、L4 LLM），没有层间集成。作者评价："屎山搬运"。

**两次失败的根本原因：** AI 选择了自己的执行策略，而不是遵循文档2中定义的顺序任务卡流程。

## 推荐环境

| 环境 | 推荐度 | 原因 |
|:---|:---|:---|
| **IDE**（Google IDX、Cursor、Windsurf 等） | ✅ **推荐** | 单 Agent 执行、可视化文件树、实时反馈、天然顺序执行 |
| **CLI**（Claude Code、终端 Agent） | ⚠️ **谨慎使用** | 多 Agent 并行、黑盒执行、完成后才能看到结果 |

**原始零错误构建是在 Google IDE + Claude Opus 4.5 环境下完成的。** IDE 环境天然约束 AI 为单线程顺序执行——正是这个架构所需要的。

两次 CLI 失败都是因为 CLI Agent 有自由启动并行子 Agent 的能力。AI 把"构建系统"理解为"同时构建所有东西"——这违反了逐层集成原则。

**如果必须使用 CLI：** 下方的启动提示词包含了防止并行的约束，但 IDE 级别的强制执行天然比提示词级别的约束更可靠。

## 正确的启动协议

### 第一步：将两份文档作为上下文加载

将文档1和文档2**同时**提供给 AI 的同一个会话。暂时不要开始编码。

### 第二步：使用以下启动提示词

复制并粘贴以下提示词开始：

```
我正在给你提供一个名为 LiteClaw 的项目的两份架构文档。

文档1（架构蓝图）定义了要造什么。
文档2（任务卡系统）定义了怎么造。

关键规则——违反任何一条都会产生损坏的代码：

1. 仅限顺序执行：严格按顺序执行任务卡：
   TASK-00 → TASK-01 → TASK-02 → ... → TASK-28。
   绝不跳跃。绝不并行。绝不批量处理多个任务。

2. 每个任务一个会话：每张任务卡 = 一次独立编码会话。
   完整完成一个任务后停下来，汇报你构建了什么。
   等待我的验收确认后再进行下一个任务。

3. 层级集成：每个新任务必须导入并引用之前已完成任务的代码。
   不允许孤立模块。每个新文件必须连接到现有代码库。

4. 禁止模板倾倒：不要生成样板代码或占位符代码。
   每一行代码都必须实现任务卡中指定的实际功能。
   如果任务卡写"最多80行"，就要严格遵守。

5. 验证后再继续：每个任务完成后，运行指定的测试。
   汇报：创建的文件、代码行数、测试结果、与前面层的集成点。
   然后等待我的确认。

6. 不要重组：严格遵循任务卡中指定的目录结构和文件名。
   不要重命名、重构或"优化"组织结构。

现在从 TASK-00 开始。给我看你构建的内容。完成 TASK-00 后停下，
等待我的验收。
```

### 第三步：每个任务验收后再继续

每个任务完成后，检查：

```
□ 创建的文件与任务卡规格一致？
□ 代码行数在任务卡限制内？
□ 所有指定测试通过？
□ 新代码导入/引用了之前层的代码？
□ 没有占位符或桩函数？
```

如果全部通过，告诉 AI：**"TASK-XX 验收通过。继续 TASK-XX+1。"**

如果任何检查失败：**"TASK-XX 验收失败。开新会话，仅重做 TASK-XX。"**
不要让 AI 在同一会话中修复——熵会累积。

### 第四步：阶段检查点

每个阶段结束时，做一次完整的集成测试：

```
Phase 0（TASK-00）：     pip install -e . 可用，原 nanobot 功能正常
Phase 1（TASK-01~04）：  安全层集成到 Agent Loop，密钥永不泄露
Phase 2（TASK-05~09）：  路由器按任务级别选择模型，故障转移正常
Phase 3（TASK-10~13）：  上下文层替换原 context.py，Token 预算生效
Phase 4（TASK-14~16）：  审计引擎拦截破坏性操作，日志写入 Obsidian
Phase 5（TASK-17~20）：  执行引擎通过统一 ExecutionManager 连接
Phase 6（TASK-21~23）：  所有数据持久化到 SQLite，成本追踪准确
Phase 7（TASK-24~28）：  UI 连接真实组件，全部5个标签页功能正常
```

## 反模式（不要这样做）

| ❌ 反模式 | 为什么会失败 | ✅ 正确做法 |
|:---|:---|:---|
| "构建完整系统" | AI 并行化所有内容，产出互不关联的模块 | "只从 TASK-00 开始。停下来等待。" |
| 先给文档1，后给文档2 | AI 在看到施工图纸前就开始编码 | 同时给两份文档，然后使用启动提示词 |
| 让 AI "优化"架构 | AI 重组文件、重命名类、添加框架 | "严格按照规格执行。不做任何修改。" |
| 在同一会话中修复失败的任务 | 会话上下文被错误代码污染，熵增加 | 开新会话，从头重做失败的任务 |
| 跳过任务间的验收 | 第 N+1 层建在有缺陷的第 N 层上，错误级联 | 验收每一个任务。无例外。 |
| 使用多个并行 Agent | 每个 Agent 相互隔离，产出的代码无法集成 | 单个顺序 Agent。一次一个任务。 |

## 验证结果（2026年3月28日）

### 三轮测试，同一天、同一 CLI 环境、同一 Opus 4.6：

| 轮次 | 启动指令 | 执行方式 | 结果 |
|:---|:---|:---|:---|
| **第1轮** | 先喂文档1，后补文档2 | AI 自行决定执行顺序 | ❌ 61个测试通过但模块孤立，无 Agent Loop |
| **第2轮** | "编程出完整的 LiteClaw 软件" | 6个并行 Agent | ❌ 模板化 UI、缺失层级、屎山 |
| **第3轮** | **"绝对禁止多Agent！按部就班！"** + 顺序执行强制 | **严格 TASK-00→28，单 Agent** | ✅ **227个测试通过，全部8层集成，完整产品** |

**同样的文档。同样的模型。同样的环境。只有启动指令改变了。**

### 第3轮验证指标：

- **29张任务卡：** 全部严格按顺序完成
- **227个测试：** 全部通过，各阶段零回归
- **8个架构层：** L0 Security → L7 Audit，通过 Agent Loop 完整集成
- **阶段检查点：** 每个阶段验证后才进入下一阶段
- **测试累积轨迹：** 6 → 34 → 80 → 91 → 118 → 167 → 196 → 227（单调递增）

### 历史验证：

- **原始构建（2026年2月）：** Google IDE + Claude Opus 4.5 → 约5小时，近乎零错误
- **CLI 构建（2026年3月28日）：** Claude Code CLI + Claude Opus 4.6 → 全部29个任务完成，227个测试通过
