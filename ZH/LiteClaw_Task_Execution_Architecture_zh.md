# LiteClaw 任务执行系统架构
> 用途：喂给 Claude Code 作为系统设计上下文  
> 版本：1.0 · 2026年2月  
> 核心原则：AI主动系统 · Skill复用 · Agent单次生命周期 · 人类作为外部观测器

---

## 0. 根本范式：从人类主动到 AI 主动

```
过去（人类主动）：
  人类发起请求 → LLM响应 → 结束
  每次对话是独立函数调用，AI是无状态工具

现在（AI主动）：
  HEARTBEAT定时触发 → AI感知环境 → 自主规划 → 执行 → 反馈循环
  AI是有状态的、时间连续的行为主体
```

**关键推论：**
- Skill 是知识结晶，**永久复用**，跨任务/跨 Agent 通用
- Agent 实例是执行上下文，**用完即删**，每次从干净状态出发
- 人类不是"用户"，是**系统稳定性维护者**和**外部独立观测器**

## 2. 任务分析层：如何解析用户输入

### 2.1 意图识别的四要素（逻辑闭环）

收到任务后，必须从输入中提取或补全这四要素，缺一不可：

```
① 现状描述  — 当前是什么状态？（现在有什么信息/资源）
② 目标定义  — 期望达到什么状态？（明确的终态）
③ 约束边界  — 什么不能做？（权限/时间/格式限制）
④ 验收标准  — 如何判断成功？（可测量的完成条件）
```

**缺少任何一环 → AI用幻觉填充 → 结果漂移**

补全方式：
- 优先从 `USER.md`、`memory/`、`AGENTS.md` 推断
- 仍不清晰 → 向用户提出**唯一最关键**的澄清问题（不要列多个问题）
- 确实无法判断 → 挂起，通知用户

### 2.2 任务分类矩阵

| 维度 | 选项 | 影响 |
|------|------|------|
| **触发方式** | 用户主动 / HEARTBEAT定时 / 事件驱动 | 决定优先级和响应时效 |
| **执行步骤** | 单步 / 多步顺序 / 多步并行 | 决定 Agent 编排方式 |
| **输出类型** | 文字/图片/视频/语音/GUI操作 | 决定调用哪些 Skill |
| **外部依赖** | 纯本地 / 需API / 需浏览器 | 决定失败率和重试策略 |
| **风险等级** | 只读 / 可逆写 / 不可逆操作 | 决定是否需要人工确认 |

### 2.3 复杂度评估与路由

```python
# 伪代码：任务路由逻辑
if 单步 and 只读 and 无外部依赖:
    → 直接执行，无需规划
elif 多步 and 可逆 and 有明确SOP:
    → 加载对应 Skill，按步骤执行
elif 涉及不可逆操作 or 超出SOUL覆盖:
    → 挂起，Telegram通知用户确认
elif 需要多个Skill协作 or 预计>10次工具调用:
    → 建议用户启用 Research/深度任务模式
```

## 3. 任务执行架构：Skill 体系

### 3.1 Skill 的本质

**Skill = 将人类 SOP 转化为 AI 可执行的结构化知识包**

```
skills/
└── skill-name/
    ├── SKILL.md      ← 核心：描述 + 步骤 + 约束 + 验收
    ├── package.json  ← 依赖（可选）
    └── scripts/      ← 执行脚本（可选）
```

**SKILL.md 必须包含的字段：**

```markdown
---
name: skill-name
description: 一句话描述（用于渐进式披露的索引）
triggers: [关键词1, 关键词2]  # AI判断何时加载此Skill
output_type: text|image|video|audio|gui
risk_level: readonly|reversible|irreversible
---

# Skill名称

## 前置条件
- 需要什么权限/工具/环境

## 执行步骤
1. 步骤一
2. 步骤二（每步包含：做什么 + 验证什么）

## 失败处理
- 步骤N失败时：降级路径 or 挂起条件

## 验收标准
- 成功的可测量条件

## 注意事项
- 已知的陷阱和边界条件
```

### 3.2 Skill 加载机制：渐进式披露

```
系统提示词中只注入 Skill 的 description（轻量索引）
                    ↓
AI识别任务需要某个 Skill
                    ↓
动态读取完整 SKILL.md 内容注入上下文
                    ↓
执行 Skill 定义的步骤
                    ↓
任务完成，Skill 上下文释放（不留在 Session）
```

**为什么这样设计：**
- 所有 Skill 一次性加载 = Token 爆炸
- 渐进式加载 = 上下文窗口只包含当前任务所需

### 3.3 当前需要构建的 Skill 清单

#### 信息采集类
```
skill-rss-collector      # RSS/Atom订阅采集
skill-web-scraper        # 网页内容抓取（调用BrowserEngine）
skill-api-fetcher        # 第三方API数据拉取
skill-dedup-filter       # 去重+相关性过滤
```

#### 内容生产类
```
skill-article-writer     # 文章生成（结构+风格对齐）
skill-video-script       # 视频脚本（分镜+口播稿+时间轴）
skill-audio-tts          # 语音合成（调用Qwen3-TTS）
skill-image-gen          # 图片生成（调用图像模型）
```

#### GUI自动化类
```
skill-web-login          # 网页登录（拟人操作）
skill-form-fill          # 表单填写
skill-content-publish    # 内容发布（各平台）
skill-data-extract       # 网页数据提取
```

#### 知识管理类
```
skill-obsidian-write     # 写入Obsidian（已验证）
skill-memory-distill     # memory蒸馏（触发人工审查）
skill-knowledge-search   # 语义检索知识库
```

## 4. MCP 接口调用架构

### 4.1 MCP 的定位

MCP（Model Context Protocol）= AI 调用外部软件的标准化接口

```
AI Agent
    │
    ├── Skill（自定义SOP，Markdown定义）
    │
    └── MCP Server（外部软件能力接入）
            ├── 文件系统 MCP
            ├── 浏览器控制 MCP（Playwright/Puppeteer）
            ├── 数据库 MCP（SQLite）
            ├── Obsidian MCP（已验证写入）
            ├── Telegram MCP（已验证通信）
            └── 自定义 MCP（DGX Spark推理接口）
```

### 4.2 MCP 调用的标准流程

```
1. 任务分析层识别需要哪个MCP能力
2. 检查MCP Server是否可达（健康检查）
3. 构建调用参数（经过 PromptFirewall 出口扫描）
4. 执行调用
5. 结果经过 PromptFirewall 入口扫描（防恶意返回内容）
6. 结果写入 Session / 存储层
7. 失败时：重试 → 降级 → 挂起（按风险等级）
```

### 4.3 关键 MCP 的特性与注意事项

#### 浏览器控制 MCP（最脆弱）
```
特性：
- 支持拟人点击、输入、滚动、截图
- 可处理动态JS渲染页面

注意事项：
- 页面DOM结构变化会导致选择器失效（无声失效！）
- 执行前必须快照当前DOM哈希作为基线
- 执行后对比验证结果
- 需要HEARTBEAT定期运行"Skill可用性探测"
- 登录状态/Cookie过期要有检测机制
- 不可逆操作（发布/删除）必须人工确认

降级路径：
- 主选择器失效 → 尝试备用选择器
- 备用也失效 → 截图+通知用户手动操作
```

#### 文件系统 MCP
```
特性：
- 读写本地文件，监控文件变更
- Obsidian Vault 实时写入（已验证）

注意事项：
- 写入前备份重要文件
- 路径权限隔离（不能越出工作目录）
- 大文件读取只取摘要+关键段，不全量载入Session
```

#### LLM 推理 MCP（DGX Spark本地）
```
特性：
- GPT-OSS 120B 本地推理，零API成本
- 适合大批量内容生产任务

注意事项：
- 显存占用大，并发请求需要队列管理
- 本地模型输出质量与API模型有差异，需要分级路由
- 任务路由规则：简单任务→本地模型，高质量输出→API模型
```

#### 语音 MCP（Qwen3 TTS/STT）
```
特性：
- TTS：文字→语音，已验证
- STT：语音→文字，已验证，支持Telegram语音消息

注意事项：
- 多音字/专有名词需要在Skill中维护自定义词典
- 长文本TTS需要分段处理（单次有长度限制）
- 情感韵律标记影响输出质量，需要在Skill中定义标记规范
```

## 5. 输出类型系统

### 5.1 输出类型判断树

```
任务收到后，首先判断输出类型：

用户需要"阅读/了解"某事？
  → 文字输出（Markdown / 纯文本 / 结构化报告）

用户需要"看到"某物？
  → 图片输出（截图 / AI生成图 / 数据可视化）

用户需要"听到"某内容？
  → 语音输出（Qwen3-TTS合成）

用户需要"发布/分发"内容？
  → 多模态组合（文字+图片+语音 按平台要求组合）

用户需要AI"代替操作"某网站/软件？
  → GUI控制输出（BrowserEngine拟人操作）
```

### 5.2 各输出类型的技术规格

#### 文字输出
```
格式：Markdown / 纯文本 / JSON / HTML
存储：Obsidian写入（带时间戳，已验证）
风格锚：SOUL.md中文定义的核心调性
质量控制：结构完整性检查 + 人工最终审阅（重要内容）
Token消耗：直接影响成本，长文本分段生成
```

#### 图片输出
```
来源：
  - 网页截图（BrowserEngine）
  - AI生成（图像模型API）
  - 数据可视化（本地生成）
格式：PNG / JPEG / WebP
存储：本地文件系统 + Obsidian附件
注意：AI生成图片需要版权标注
```

#### 视频输出
```
当前阶段：视频脚本生成（文字），不直接生成视频文件
脚本格式：
  - 分镜描述
  - 口播稿（标注语速/停顿）
  - 时间轴（每段时长）
  - 配乐建议
未来扩展：接入视频生成API（Sora/Kling等MCP）
```

#### 语音输出
```
引擎：Qwen3-TTS（已验证）
触发：文字内容 → TTS Skill → 音频文件
发送：Telegram语音消息
注意事项：
  - 单次处理上限：约1000字
  - 超长内容：分段生成+合并
  - 专有名词：维护自定义发音词典
```

#### GUI控制输出
```
本质：AI观察屏幕/页面 → 规划操作序列 → 执行拟人操作
操作类型：
  - 点击（按钮/链接/菜单）
  - 输入（文本框/搜索框）
  - 滚动（加载更多内容）
  - 等待（动态加载完成）
  - 截图（验证操作结果）
安全规则：
  - 执行前快照DOM哈希
  - 不可逆操作必须人工确认
  - 每步操作后验证预期状态
  - 失败时截图留证
```

## 6. 任务启动机制

### 6.1 三种启动方式

#### 方式一：用户主动触发
```
输入渠道：
  - Telegram消息（文字/语音/图片）
  - Web UI（Gradio 9标签页）
  - 语音输入（STT转文字后处理）

处理流程：
  用户输入 → STT（如果是语音）→ 意图识别
  → 四要素提取 → 任务规划 → 执行 → 输出
```

#### 方式二：HEARTBEAT 定时触发
```
触发源（HEARTBEAT.md中定义）：
  - interval：固定间隔（每N分钟/小时）
  - cron：精确时间点（每天08:00）
  - threshold：条件触发（信息量超过N条）
  - event：外部事件（文件变更/API回调）

防抖机制：250ms合并窗口（多源同时触发只执行一次）
重复抑制：相同任务在冷却期内不重复执行
```

#### 方式三：级联触发（任务链）
```
上游任务完成 → 检查是否触发下游任务

示例：
  RSS采集完成（50条新内容）
    → 触发过滤Skill
    → 过滤后剩15条高相关内容
    → 触发摘要生成Skill
    → 生成日报
    → 触发TTS Skill（如果配置了语音推送）
    → 发送Telegram语音消息
```

### 6.2 任务队列管理

```
优先级：
  P0 - 用户直接输入（实时响应）
  P1 - 不可逆操作的人工确认回调
  P2 - HEARTBEAT定时任务
  P3 - 级联触发的后续任务
  P4 - 后台知识整理任务

串行规则：
  - 同一Session内的操作必须串行（防竞态）
  - 不同Session的只读操作可并行
  - 涉及同一外部资源的写操作必须串行+加锁
```

## 7. 任务执行中的关键注意事项

### 7.1 信噪比原则（最重要）

**人类输入质量直接决定AI输出质量**

高质量输入 = 逻辑闭环的四要素 → AI输出空间收敛 → 结果可预期  
低质量输入 = 缺要素/歧义/矛盾 → AI幻觉填充 → 结果漂移

**设计原则：**
- 与用户的交互界面要引导结构化输入
- Telegram消息模板（可选）降低输入噪声
- 歧义时提一个最关键问题，不要列清单

### 7.2 Session Memory 的 Token 管理

```
Session是Token天坑，必须主动管理：

热层（保留完整）：最近5-8轮对话
温层（压缩保留）：中间段落只保留"动作+结论"，去掉推理过程
冷层（摘要归档）：早期内容摘要后写入memory/，Session删除

特别注意：
- 工具调用结果不要全量保留在Session（只保留关键值）
- 失败重试的推理过程折叠为一句话
- SOUL/IDENTITY启用Prompt Caching（固定内容不重复计费）
- 超过N轮时HEARTBEAT触发主动压缩
```

### 7.3 幂等性设计

**所有 Skill 默认设计为幂等（重复执行不产生副作用）**

```
非幂等操作（必须特殊处理）：
  - 发送消息/邮件 → 加发送记录，重复前检查
  - 创建文件 → 检查是否已存在
  - API写操作 → 使用幂等键（idempotency key）
  - 数据库写入 → upsert而非insert

不可逆操作（必须人工确认）：
  - 删除文件/数据
  - 发布到公开平台
  - 发送给真实用户的消息
  - 支付/订单操作
```

### 7.4 持续学习的稳定性风险

```
风险一：覆盖盲区
  SOUL只覆盖预见到的情境
  对策：SOUL加"元规则"——遇到未覆盖情境先挂起

风险二：信息熵增
  memory/规则随时间过时
  对策：每条记忆加时间戳+有效期，定期保鲜度检查

风险三：自我强化失稳
  AI把某次成功路径固化，新情境下仍强行复用
  对策：memory写入加置信度标注，成功路径需人工确认才固化

风险四：Agent漂移
  长期运行的Agent因Session积累偏离SOUL设定
  对策：Agent单次生命周期，每次从干净状态出发
```

## 8. 结果验证与任务关闭

### 8.1 结果验证的三个层次

```
层次一：格式验证（自动）
  输出是否符合预期格式？
  文件是否成功创建？
  API是否返回200？

层次二：语义验证（LLM辅助）
  输出内容是否回答了原始问题？
  是否满足四要素中的验收标准？

层次三：业务验证（人工）
  结果是否真正有用？
  是否存在事实错误？
  是否需要进一步迭代？
```

### 8.2 任务关闭流程

```
1. 结果验证通过
2. 输出交付（发送到Telegram / 写入Obsidian / 存储层归档）
3. 有价值的洞察写入memory/（带时间戳+置信度）
4. 更新任务执行统计（成本/耗时/工具调用次数）
5. Session Memory清理（冷层摘要归档）
6. Agent实例销毁
```

### 8.3 四级修复响应

```
L1 自动修复：Skill内置重试，幂等操作自动重执行
L2 降级处理：主路径失败→备用路径，复杂→简单，输出降级通知
L3 挂起等待：高风险/超出SOUL范围→AI主动暂停+Telegram通知
L4 全局回滚：系统性漂移→恢复最近人工确认的稳定状态（Git）
```

## 9. HEARTBEAT.md 设计模板

```markdown
# HEARTBEAT 心跳配置

## 定时任务
- cron: "0 8 * * *"     → skill-daily-briefing      # 每天08:00日报
- cron: "0 */6 * * *"   → skill-rss-collector        # 每6小时采集
- cron: "0 9 * * 1"     → skill-knowledge-freshness  # 每周一知识保鲜检查
- cron: "0 */1 * * *"   → skill-health-check         # 每小时健康检查

## 阈值触发
- when: new_items > 20  → skill-dedup-filter         # 新内容超20条触发过滤
- when: session_turns > 15 → skill-session-compress  # Session超15轮触发压缩

## 事件触发
- on: obsidian_change   → skill-knowledge-index      # Vault变更触发索引更新
- on: skill_failure > 3 → notify_user               # Skill连续失败3次通知

## 防抖设置
- merge_window: 250ms   # 多源触发合并窗口
- cooldown: 300s        # 同类任务冷却时间
```

## 10. 任务类型示例：端到端流程

### 示例一：信息采集→文章生成

```
触发：HEARTBEAT cron 每天08:00

步骤：
1. skill-rss-collector
   → 拉取订阅源，获取原始内容列表
   → 输出：raw_items[]

2. skill-dedup-filter
   → 去重（SimHash） + 相关性评分 + 时效性判断
   → 输出：filtered_items[]（高相关内容）

3. skill-knowledge-search
   → 检索历史memory/，补充背景知识
   → 输出：context[]

4. skill-article-writer
   → 基于filtered_items + context 生成文章
   → 输出：article.md（写入Obsidian）

5. 结果验证：
   → 格式检查（Markdown结构完整）
   → 字数/质量自动评估
   → Telegram通知用户"今日文章已生成，待审阅"

6. 人工审阅节点（不可跳过）
   → 用户回复"发布" → 触发 skill-content-publish
   → 用户回复"修改建议" → 触发修改循环
```

### 示例二：语音输入→网页操作

```
触发：用户发送Telegram语音消息

步骤：
1. STT（Qwen3）
   → 语音转文字
   → 输出："帮我在XX平台发布这篇文章"

2. 意图识别
   → 任务类型：GUI控制
   → 风险等级：不可逆（公开发布）
   → 需要：文章内容（从Obsidian检索）

3. 人工确认节点（不可逆操作）
   → 发送预览到Telegram："将发布以下内容，确认请回复'确认'"
   → 等待用户确认

4. skill-content-publish（用户确认后）
   → BrowserEngine启动
   → 执行前快照DOM哈希
   → 登录检查（Cookie有效性）
   → 拟人操作：点击发布入口→粘贴内容→填写标签→点击发布
   → 执行后截图验证发布成功
   → 输出：发布成功URL

5. 结果写入
   → Obsidian记录发布时间+URL
   → Telegram回复成功截图
```

### 示例三：HEARTBEAT健康检查

```
触发：HEARTBEAT 每小时

检查项：
1. Gateway端口可达？
2. LLM API响应时间 < 5s？
3. DGX Spark本地推理可用？
4. Telegram Bot心跳正常？
5. 过去1小时是否有L3级挂起未处理？
6. Session平均Token消耗是否超阈值？

结果处理：
  全部正常 → 静默（不打扰用户）
  单项异常 → 记录日志，等待下次检查确认
  连续2次异常 → Telegram通知用户
  关键服务不可达 → 立即通知 + 触发降级模式
```

## 附录：关键设计原则速查

| 原则 | 一句话 | 反例 |
|------|--------|------|
| Skill复用 | Skill是函数库，永久存在 | 每次任务重写逻辑 |
| Agent单次 | Agent是进程，用完即删 | 让Agent长期运行积累状态 |
| 渐进披露 | 只加载当前需要的Skill | 所有Skill一次性注入 |
| 幂等优先 | 重复执行不产生副作用 | 没有重复检查的写操作 |
| 不可逆必确认 | 发布/删除必须人工节点 | 自动执行不可逆操作 |
| 热温冷裁减 | Session不超过N轮完整保留 | Session无限增长到爆炸 |
| 外部观测器 | 人类始终保有最终控制权 | 完全自动化无人工节点 |
| 中文SOUL | 母语定义人格，信息密度最高 | 英文复制粘贴行为准则 |
| 置信度标注 | 成功经验不自动固化 | AI自我强化错误路径 |
| 有效期记忆 | memory内容有TTL，定期保鲜 | 永远相信过期的记忆 |
```
